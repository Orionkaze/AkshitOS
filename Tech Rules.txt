AKSHITOS – TECH RULES & STACK DEFINITION

Version: 1.0
Scope: SaaS Edition (Authenticated, DB-backed)

1. CORE STACK (NON-NEGOTIABLE)

These technologies are locked for v1.0.

1.1 Framework

Next.js (App Router only)

TypeScript mandatory

No Pages Router

No mixing routing paradigms

Reason:
App Router provides better server component control and future scalability.

1.2 Language

TypeScript strict mode enabled

No any unless absolutely necessary

All API responses typed

All Prisma models mapped to types

Reason:
Prevents silent bugs and enforces discipline.

1.3 Styling

Tailwind CSS only

No random CSS files

No inline styles unless dynamic necessity

No mixing with Bootstrap or MUI

Design tokens must be centralized in Tailwind config.

1.4 Authentication

NextAuth.js

JWT session strategy

Prisma adapter

Email/Password + Google OAuth

No custom auth implementation.

1.5 Database

PostgreSQL

Prisma ORM

UUID for primary keys

No raw SQL unless performance-critical

All DB access through Prisma client abstraction.

2. ARCHITECTURAL RULES

AkshitOS follows:

Presentation Layer
Application Layer
Data Layer

No layer violations.

2.1 Presentation Layer Rules

UI components must not contain DB logic

No direct API fetch inside deeply nested UI

Server Components preferred

Client Components only when:

Using hooks

Using timers

Using browser APIs

2.2 Application Layer Rules (API Routes)

Every API route must:

Validate session

Validate input

Scope by userId

Execute Prisma query

Return typed JSON response

No business logic duplication across routes.

2.3 Data Layer Rules

Single Prisma instance (singleton pattern)

No multiple DB clients

Migrations must be version controlled

No schema changes without migration

3. AUTH RULES
3.1 Session Rules

All protected routes must check server session

No client-only protection

API routes must not trust client-sent userId

UserId must come from session only.

3.2 Password Rules

Bcrypt hashing

No plaintext storage

No logging passwords

No weak validation

3.3 OAuth Rules

Secure callback URLs

Environment variables required

Production keys separate from dev

4. DATABASE RULES
4.1 Schema Discipline

All models must include:

id (UUID)

userId

createdAt

updatedAt (if mutable)

No table without user ownership.

4.2 Query Discipline

Every query must include:

where: { userId: session.user.id }

No global queries.

4.3 Performance Rules

Add indexes on userId

Add index on date fields where used in sorting

Avoid N+1 queries

Avoid unnecessary includes

5. STATE MANAGEMENT RULES

Local state per module

No global state unless absolutely necessary

React Context only for:

Theme

User session (if needed)

No Redux for v1

Derived data must use useMemo.

6. TIMER ENGINEERING RULES (FocusForge)

Interval stored in useRef

Clear interval on unmount

Prevent duplicate saves

Persist only completed sessions

No infinite loops

7. CALCULATION RULES

All calculation logic must be isolated in:

/lib/calculations.ts

No math logic repeated in multiple components.

Examples:

Savings calculation

Study aggregation

Productivity trend

Resume scoring

8. VALIDATION RULES

All forms must:

Validate required fields

Validate numeric input

Prevent negative values

Sanitize strings

Validation must exist in:

Frontend

Backend

Backend validation is mandatory.

9. ERROR HANDLING RULES

Backend must return structured responses:

{
success: false,
message: string
}

Frontend must:

Display toast

Not silently fail

Not expose stack traces

10. PERFORMANCE RULES

Target:

Initial load < 2 seconds

API response < 500ms

Rules:

No heavy computation in render

Memoize derived data

Lazy load heavy modules if needed

No unnecessary re-renders

11. COMPONENT RULES

Max 200–250 lines per component

Extract reusable parts

No duplicated UI logic

Clear prop typing

No anonymous huge inline functions

12. CODE STYLE RULES

ESLint enabled

Prettier enforced

No unused imports

No commented dead code

Clear naming conventions

Naming:

Components → PascalCase
Variables → camelCase
Constants → UPPER_SNAKE_CASE
Folders → lowercase

No vague names like:

data1

finalFinal

tempFix

13. GIT WORKFLOW RULES

Branches:

main → production
dev → integration
feature/{module}

Commit format:

feat: add devmood POST route
fix: resolve streak reset bug
refactor: extract validation logic

No:
“update”
“done”
“fixing stuff”

14. DEPLOYMENT RULES

Environment variables required:

DATABASE_URL
NEXTAUTH_SECRET
GOOGLE_CLIENT_ID
GOOGLE_CLIENT_SECRET

No secrets hardcoded.

Production checklist:

Build success

Migrations applied

OAuth tested

Auth tested

DB tested

15. SECURITY RULES

No exposing sensitive data in API response

No trusting client-side values

Sanitize resume input

Validate request body length

Protect against simple injection

16. SCALABILITY RULES

Code must assume:

Backend may expand

Multi-role may exist later

Real analytics may be added

API versioning may occur

Avoid hardcoded logic that blocks future scaling.

17. OUT-OF-SCOPE RULES

Do NOT implement in v1:

Role-based access

Realtime websockets

AI APIs

Microservices

Payment systems

Complex caching systems

Stay disciplined.

18. DEFINITION OF TECH COMPLETE

Tech layer is considered complete when:

Auth stable

All modules DB-backed

No data leaks

No console errors

No unhandled promises

Clean commit history

Production deployment successful
